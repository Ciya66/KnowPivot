## 第一步：实体与关系分析

根据文档中的业务流程与架构设计，识别出以下核心实体及关系：

### 1. 核心实体
*   **用户**：系统的使用者，拥有积分配额。
*   **知识库**：知识的容器，包含配置信息（切片大小、相似度阈值）。
*   **文档**：上传的原始文件，具有版本和解析状态。
*   **文档切片**：文档解析后的文本块，对应向量存储中的 Entry。
*   **会话**：用户与 Agent 的交互上下文，属于某个知识库范围。
*   **消息**：会话中的具体对话记录（用户问题或 AI 回复）。
*   **提示词模板**：用于动态组装 Prompt 的模板对象。
*   **Token 交易记录**：记录用户积分的消耗与充值流水。

### 2. 实体关系描述
*   **用户 - 知识库**：多对多（M:N）。一个用户可以加入多个知识库，一个知识库可以有多个成员（通过 `t_kb_member` 关联表实现）。
*   **知识库 - 文档**：一对多（1:N）。一个知识库包含多个文档。
*   **文档 - 切片**：一对多（1:N）。一个文档解析后生成多个切片。
*   **用户 - 会话**：一对多（1:N）。一个用户可以发起多个会话。
*   **会话 - 消息**：一对多（1:N）。一个会话包含多轮对话消息。
*   **会话 - 知识库**：多对一（N:1）。一个会话通常关联一个特定的知识库上下文。

## 第四步：设计考量与优化建议

### 1. 范式与反范式设计
*   **冗余设计**：
    *   `t_document_segment` 表中冗余了 `kb_id` 字段。虽然可以通过 `doc_id` 关联查询得到，但在进行知识库级别的向量检索元数据过滤时（如“查找知识库 A 下的切片”），直接通过 `kb_id` 过滤能极大提升查询效率，避免多表联表查询（ESR 原则）。
    *   `t_message` 表中存储了 `token_count`。虽然可以通过流水表汇总，但为了快速展示会话消耗和历史记录，进行了冗余存储。
    *   `t_conversation` 表中冗余了 `last_message_time`。这是为了支持“最近会话列表”的高效排序，避免扫描 `t_message` 表。

### 2. 性能瓶颈预测与应对
*   **大表预警**：
    *   `t_message` 和 `t_document_segment` 将会成为数据量最大的表。
*   **优化策略**：
    *   **分表策略**：建议对 `t_message` 按 `conversation_id` 或时间进行分表（如按月分表）。对于 `t_document_segment`，由于其主要用于元数据查询，如果数据量达到亿级，建议引入 Elasticsearch 进行元数据检索，MySQL 仅作为持久化存储。
    *   **归档策略**：对于超过 6 个月的对话记录，建议迁移至冷存储或进行归档处理，保持热库轻量。
    *   **JSON 字段**：`t_message` 的 `references` 字段使用了 JSON 类型。MySQL 8.0 对 JSON 支持较好，但如果查询需求增加（如“搜索引用了某文档的所有消息”），JSON 字段的索引效率较低，建议后续考虑将引用关系抽离为独立的关联表。

### 3. 扩展性
*   **向量引擎切换**：当前设计通过 `index_name` 关联 Redis。如果未来需要迁移至 Milvus 或 Elasticsearch，只需修改写入和检索的 Infrastructure 层代码，数据库表结构无需大规模变更。
*   **多模态支持**：`t_document` 和 `t_message` 的结构较为通用。如果未来支持图片或音频，可在 `t_message` 中增加 `media_url` 字段，或在 `content` 中存储多模态 JSON 结构，扩展性良好。

### 4. 安全性建议
*   **敏感数据**：`t_user` 表中的 `password_hash` 必须使用强哈希算法（如 BCrypt）。虽然手机号等字段在当前需求未显式列出，若后续扩展，需对手机号、邮箱等字段考虑加密存储或脱敏展示。
*   **逻辑删除**：所有核心业务表均配置了 `is_deleted`。这对于审计恢复非常重要，特别是在 Agent 误删除知识库或文档的场景下，可进行数据找回。
