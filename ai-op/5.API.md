# 知枢 Agentic 智能知识库平台 - 接口设计文档

## 第一步：业务梳理

根据需求文档与数据库设计，识别出以下核心业务实体及其操作属性：

| 业务实体   | 对应数据表         | 核心操作 (CRUD)                 | 关键字段映射                                   | 敏感字段处理                                                       |
| :--------- | :----------------- | :------------------------------ | :--------------------------------------------- | :----------------------------------------------------------------- |
| **用户**   | `t_user`           | 新增(注册), 查询(登录/信息)     | `username`, `nickname`, `token_quota`          | `password_hash` 严禁返回；`avatar_url` 需拼接完整访问路径。        |
| **知识库** | `t_knowledge_base` | 增, 删, 改, 查                  | `name`, `description`, `config` (JSON->Object) | 无特殊敏感字段，需做好权限校验（`creator_id`）。                   |
| **文档**   | `t_document`       | 上传(增), 查询(列表/状态), 删除 | `file_name`, `status`, `chunk_count`           | `storage_path` 不建议直接返回 MinIO 内部路径，可返回临时下载链接。 |
| **会话**   | `t_conversation`   | 新建, 查询(列表), 删除          | `title`, `last_message_time`                   | 无。                                                               |
| **消息**   | `t_message`        | 查询(历史), 新增(发送)          | `role`, `content`, `references` (JSON->Object) | 无。                                                               |

---

## 第二步：接口定义概览

遵循 RESTful 风格，API 路径前缀统一为 `/api/v1`。根据架构设计的模块划分，接口分组如下：

| 模块                   | 路径前缀                                   | 说明                           |
| :--------------------- | :----------------------------------------- | :----------------------------- |
| **模块一：用户与认证** | `/api/v1/auth`, `/api/v1/users`            | 处理注册、登录、用户信息获取。 |
| **模块二：知识库管理** | `/api/v1/knowledge-bases`                  | 知识库的 CRUD 及成员管理。     |
| **模块三：文档管理**   | `/api/v1/knowledge-bases/{kbId}/documents` | 文档的上传、状态查询及管理。   |
| **模块四：智能对话**   | `/api/v1/conversations`, `/api/v1/chat`    | 会话管理及核心问答交互。       |

## 第三步：详细接口设计

### 模块一：用户与认证模块

#### 1. 用户注册

- **接口名称**：用户注册
- **接口地址**：`POST /api/v1/auth/register`
- **请求方式**：POST
- **接口说明**：新用户注册，密码需加盐哈希存储。

**请求参数**

| 参数名   | 类型   | 位置 | 必填 | 长度限制 | 说明                        |
| :------- | :----- | :--- | :--- | :------- | :-------------------------- |
| username | String | Body | 是   | 4-64     | 用户名，唯一                |
| password | String | Body | 是   | 8-32     | 密码明文，传输需 HTTPS 加密 |
| nickname | String | Body | 否   | 2-64     | 昵称，默认同用户名          |

**响应数据**

```json
{
  "code": 200,
  "message": "注册成功",
  "data": {
    "userId": "182837465678902345" // 雪花算法生成的ID
  },
  "timestamp": "2023-10-27 10:00:00"
}
```

**状态码说明**

- 200：成功。
- 400：参数格式错误。
- 409：用户名已存在。

- **接口名称**：用户登录
- **接口地址**：`POST /api/v1/auth/login`

**请求参数**

| 参数名   | 类型   | 位置 | 必填 | 说明   |
| :------- | :----- | :--- | :--- | :----- |
| username | String | Body | 是   | 用户名 |
| password | String | Body | 是   | 密码   |

**响应数据**

```json
{
  "code": 200,
  "message": "登录成功",
  "data": {
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "tokenType": "Bearer",
    "expiresIn": 7200,
    "userInfo": {
      "userId": "182837465678902345",
```

**状态码说明**

- **接口名称**：上传文档
- **接口地址**：`POST /api/v1/knowledge-bases/{kbId}/documents/upload`
- **请求方式**：POST
- **Content-Type**：`multipart/form-data`

**请求参数**

| 参数名   | 类型   | 位置 | 必填 | 说明                       |
| :------- | :----- | :--- | :--- | :------------------------- |
| kbId     | Long   | Path | 是   | 知识库 ID                  |
| file     | File   | Body | 是   | 文件流，支持 pdf/docx/txt  |
| fileName | String | Body | 否   | 文件名（默认取上传文件名） |

**响应数据**

```json
{
  "code": 200,
  "message": "上传成功，正在解析中",
  "data": {
    "docId": "182837465678902457",
    "status": 0, // 0:已上传
    "createTime": "2023-10-27 10:20:00"
  },
  "timestamp": "..."
}
```

_注意：上传成功后，后端会发送消息至 Kafka 进行异步解析。_

#### 2. 发送消息 (SSE 流式响应)

- **接口名称**：发送消息并获取回答
- **接口地址**：`POST /api/v1/chat/messages`
- **请求方式**：POST
- **Accept**：`text/event-stream`

**请求参数**

| 参数名         | 类型   | 位置 | 必填 | 说明         |
| :------------- | :----- | :--- | :--- | :----------- |
| conversationId | Long   | Body | 是   | 会话 ID      |
| content        | String | Body | 是   | 用户提问内容 |

**响应数据 (SSE 事件流)**
接口返回 `Content-Type: text/event-stream`。前端需监听事件流。

**事件定义**：

1.  **message**：携带部分回答内容。
2.  **references**：携带引用来源。
3.  **done**：回答结束，携带 Token 消耗统计。

**事件示例**：

```text
// 事件1：开始回答
event: message
data: {"delta": "根据系统架构设计文档，"}

// 事件2：继续回答
event: message
data: {"delta": "该项目采用 DDD 分层架构..."}

// 事件3：引用来源（可选，视 RAG 结果而定）
event: references
data: {"sources": [{"docName": "架构设计.pdf", "segmentId": "xxx", "content": "..."}]}

// 事件4：结束
event: done
data: {"messageId": "182837...", "tokenCount": 150, "finishReason": "stop"}
```

**错误状态码**

- 50001：Token 配额不足。
- 50002：知识库索引异常。
- 50301：模型服务不可用。

## 附录：通用响应结构与状态码定义

### 1. 统一响应结构

```json
{
  "code": 200, // 业务状态码
  "message": "操作成功", // 提示信息
  "data": {}, // 业务数据
  "timestamp": "2023-10-27 10:00:00" // 时间戳
}
```

### 2. 业务状态码规范

| 状态码    | 说明           | 场景                         |
| :-------- | :------------- | :--------------------------- |
| **200**   | 成功           | 请求处理成功。               |
| **400**   | 参数错误       | 缺少必填参数、格式错误。     |
| **401**   | 未认证         | 未登录或 Token 过期。        |
| **403**   | 无权限         | 无权访问该知识库或文档。     |
| **404**   | 资源不存在     | ID 对应的数据不存在。        |
| **409**   | 冲突           | 用户名重复、文档名重复。     |
| **429**   | 请求过多       | 触发限流保护。               |
| **500**   | 服务器内部错误 | 空指针、数据库异常。         |
| **50001** | 配额不足       | Token 余额不足。             |
| **50002** | 业务处理失败   | 文档解析失败、向量入库失败。 |
