# 知枢 Agentic 智能知识库平台 - 需求拆解与任务分工文档

## 第一步：模块化梳理与优先级定义

基于 MVP（最小可行性产品）原则，确定以下模块优先级：

| 模块编号 | 模块名称                   | 优先级 | 说明                                                               |
| :------- | :------------------------- | :----- | :----------------------------------------------------------------- |
| **M1**   | **基础架构与数据层**       | **P0** | 系统的地基，包括 DB 设计、中间件搭建、基础实体类。                 |
| **M2**   | **知识库管理（文档入库）** | **P0** | 核心链路：文档上传 -> 解析 -> 向量化 -> 入库。没有数据就没有问答。 |
| **M3**   | **智能对话核心**           | **P0** | 核心链路：问答交互 -> RAG 检索 -> LLM 生成 -> 结果返回。           |
| **M4**   | **前端交互界面**           | **P0** | 用户触点：上传界面、对话窗口。                                     |
| **M5**   | **用户权限与配额**         | **P1** | 业务支撑：鉴权、Token 配额控制。                                   |
| **M6**   | **监控与非功能需求**       | **P1** | 埋点、日志、性能优化。                                             |

## 关键风险与依赖说明

1.  **技术依赖风险**：Redis Vector Search 的性能在大规模数据（千万级向量）下可能衰减。建议 V1.0 限制单知识库文档数量，V2.0 考虑迁移至 Milvus 或 Qdrant。
2.  **模型依赖**：Python 端需提前测试 Embedding 模型与 LLM 的兼容性，确保 Token 计算准确，防止配额扣减错误。
3.  **异步处理链路**：文档入库链路较长（Java->Kafka->Python->Redis），需在 M2 阶段重点测试异常重试机制，避免消息丢失导致文档卡在“处理中”状态。

此拆解方案覆盖了 V1.0 版本的核心闭环，团队可依据此表进行 Sprint 规划与任务认领。
