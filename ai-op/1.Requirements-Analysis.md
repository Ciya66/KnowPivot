# 知枢 Agentic 智能知识库平台 - 需求分析文档

## 1. 项目定位（Agent 在系统中的角色）

在本系统中，Agent 不仅仅是问答机器人，而是**业务流程的编排者与执行者**。
- **角色定义**：智能知识助理。
- **核心职责**：
  - **意图识别**：理解用户自然语言背后的真实需求（如：查询、总结、分析、操作）。
  - **知识检索**：替代传统关键词搜索，进行语义向量检索与重排序。
  - **任务规划**：对于复杂问题，拆解为多个子步骤执行。
  - **工具调用**：具备访问外部数据源或执行特定业务逻辑（如生成报表）的能力。

## 2. 用户与使用场景

### 2.1 用户触发 Agent 方式
- **Web/移动端对话框**：用户在界面上输入自然语言（如“帮我整理上周关于A项目的会议纪要”）。
- **文档内交互**：用户在阅读文档时，选中段落触发“解释”、“翻译”或“关联查找”。
- **API 触发**：第三方系统（如企业微信、钉钉）通过 Webhook 调用 Agent 进行自动回复。

### 2.2 业务流程参与
- **入职培训场景**：新员工提问“请假流程是什么”，Agent 检索知识库，结合最新版制度文档回答，并附上流程图链接。
- **跨文档分析场景**：管理层提问“对比Q1和Q2的销售数据”，Agent 调用工具提取两个文档数据，生成对比分析摘要。

## 3. 系统整体流程（调用链路）

以下以**“用户发起复杂问答请求”**为例：

1. **用户请求接入**
   - 用户在 Vue 前端输入查询：“帮我总结一下《项目A》的需求文档，并列出潜在风险。”
   - 请求发送至 **Java 网关**。

2. **Java 业务层预处理**
   - **鉴权与风控**：验证用户 Token，检查是否有访问《项目A》的权限。
   - **上下文构造**：从 MySQL/Redis 获取用户的近期会话历史。
   - **请求转发**：Java 将用户 Query、权限过滤后的知识库 ID 列表、会话 ID 封装，通过 HTTP 异步调用 **Python Agent 服务**。

3. **Python Agent 思考与执行**
   - **Prompt 组装**：FastAPI 服务接收请求，加载 System Prompt，注入用户历史记忆。
   - **意图规划**：LLM 判断这是一个“多步任务”。
     - Step 1: 检索工具 -> 查找《项目A》需求文档。
     - Step 2: 总结工具 -> 提取核心需求。
     - Step 3: 推理工具 -> 基于需求推断风险。
   - **RAG 检索**：Agent 调用向量库，检索相关切片。
   - **响应生成**：LLM 基于检索内容生成最终答案。

4. **结果回传与持久化**
   - Python 返回 JSON 结构化数据（答案 + 引用来源 + 消耗 Token）。
   - Java 接收结果，**保存对话记录至 MySQL**，**更新用户积分/配额**，并将最终结果推送给前端。

## 4. 功能模块拆解

### 4.1 用户端模块（Vue3）
- **知识库管理 UI**：文档上传、版本回滚、权限配置。
- **智能对话窗口**：支持流式输出、引用跳转、点赞/点踩反馈。

### 4.2 Java 业务服务模块
- **文档处理引擎**：负责文档解析（Apache Tika/PDFBox）、分片、清洗，并将文本块发送给 Python 进行 Embedding。
- **向量索引管理**：利用 Redis（Search 模块）存储向量索引，维护 `doc_id` 与业务元数据的映射关系。
- **对话管理**：会话列表管理、历史记录存储。

### 4.3 Python Agent 服务模块
- **Agent 核心引擎**：基于 LangChain/LlamaIndex 构建，负责 ReAct 流程编排。
- **Prompt 工厂**：动态加载和渲染 Prompt Template。
- **工具库**：封装搜索、数据库查询、网络搜索等原子能力。

### 4.4 知识库模块
- **向量存储**：Redis Vector Store（索引创建、KNN 检索）。
- **BM25 检索**（可选）：针对关键词精确匹配的补充。

## 5. Agent 能力拆解（重点）

### 5.1 Prompt 管理
- **动态注入**：根据用户角色动态注入 System Prompt（如：管理员可见隐藏字段，普通用户不可见）。
- **Few-Shot 样库**：在 Prompt 中预置优质问答对，引导 LLM 输出格式。

### 5.2 工具调用
- **知识库检索工具**：
  - Input: `query_text`, `kb_ids`
  - Output: `context_chunks`
- **元数据过滤工具**：
  - Input: `time_range`, `author`
  - Output: SQL Filter condition (用于 Java 端过滤)

### 5.3 记忆
- **短期记忆**：FastAPI 利用 Redis 存储当前 Session 的最近 N 轮对话。
- **长期记忆**：重要对话总结后存入向量库，支持“记得我上周问过什么”。

### 5.4 知识检索（RAG）
- **混合检索**：向量检索（语义）+ 关键词检索（精确）。
- **重排序**：Python 端获取初步结果后，使用 BGE-Reranker 模型对结果进行精排，提升相关性。

## 6. 接口设计（具体规范）

### 6.1 Java → Python 接口定义
**接口路径**：`POST http://ai-service:8000/api/v1/agent/run`

**请求头**：
```json
{
  "X-Api-Key": "internal_secret_key",
  "Content-Type": "application/json"
}
```

**输入参数**：
```json
{
  "session_id": "uuid-12345",
  "user_id": "user_001",
  "query": "帮我总结项目A的风险点",
  "context": {
    "kb_ids": ["kb_01", "kb_02"],
    "role": "admin",
    "history": [
      {"role": "user", "content": "打开项目A文档"},
      {"role": "assistant", "content": "已为您找到项目A文档..."}
    ]
  },
  "config": {
    "model": "qwen-72b-chat",
    "temperature": 0.7,
    "stream": true
  }
}
```

**输出结构**：
采用 **SSE (Server-Sent Events)** 流式返回。

Event: `token`
```json
{
  "type": "token",
  "content": "根据..."
}
```
Event: `tool_call`
```json
{
  "type": "tool_call",
  "tool_name": "vector_search",
  "status": "success"
}
```
Event: `finish`
```json
{
  "type": "finish",
  "metadata": {
    "total_tokens": 1500,
    "references": [
      {"doc_name": "项目A需求.pdf", "page": 5, "text": "..."}
    ]
  }
}
```

### 6.2 错误码设计
| 错误码 | 含义 | Java 处理策略 |
| :--- | :--- | :--- |
| 20001 | 模型服务超时 | 提示用户“思考时间过长，请稍后重试” |
| 20002 | 内容安全合规拦截 | 返回“您的问题包含敏感内容，无法回答” |
| 20003 | 知识库检索为空 | Agent 触发“兜底回复”机制 |
| 50001 | Token 配额不足 | 引导用户充值或联系管理员 |

## 7. 非功能需求（AI 系统特有）

### 7.1 响应延迟优化
- **首字延迟 (TTFT)**：控制在 1.5s 以内。策略：Prompt 缓存、流式输出。
- **检索延迟**：Redis 向量检索控制在 100ms 以内。

### 7.2 Token 成本控制
- **分片限制**：文档切片上限设为 500 tokens，避免单次检索上下文过长。
- **历史压缩**：对话历史超过 5 轮时，调用 LLM 生成摘要，减少 Prompt 长度。

### 7.3 并发处理
- Python 端采用 **Async FastAPI** 支持高并发 I/O 等待。
- 限制单个用户的并发 Agent 请求数为 3，防止资源滥用。

### 7.4 缓存策略
- **Embedding 缓存**：相同文档内容不重复计算向量。
- **Query 缓存**：对完全相同的 Query（如“你好”），直接返回缓存结果，不调用 LLM。

## 8. 技术架构建议

### 8.1 为什么 Java + Python 分层？
- **Java (主系统)**：强项在于企业级业务逻辑、事务控制、权限管理、生态中间件成熟。适合做“大门”和“账房”。
- **Python (AI 服务)**：强项在于 AI 生态、快速迭代、动态类型适合处理 Prompt 模板和复杂数据结构。适合做“大脑”。
- **解耦优势**：AI 模型更新快，Python 服务可独立部署、灰度发布，不影响 Java 核心业务的稳定性。

### 8.2 消息队列与异步化
- **文档入库流程**：必须引入消息队列。
  - 用户上传 -> Java 写入 MinIO -> 发送消息到 Kafka -> Python 消费者解析文档 -> 生成向量 -> 写入 Redis。
  - **理由**：文档解析和 Embedding 过程耗时（可能数分钟），不能阻塞 HTTP 请求。
- **对话流程**：保持同步 HTTP/SSE 连接，暂不需要 MQ，除非是后台生成报告类任务。

## 9. 风险分析

| 风险类型 | 具体表现 | 缓解措施 |
| :--- | :--- | :--- |
| **模型幻觉** | 编造不存在的文档内容 | 强制 RAG 约束（Prompt: "仅根据检索内容回答"）；增加引用溯源展示。 |
| **数据隐私** | 私有数据被模型“记住”泄露 | 使用私有化部署模型；上下文窗口设置滑动窗口，不持久化敏感上下文。 |
| **成本不可控** | 用户大量调用导致 API 费用激增 | Java 层实现用户配额限制；对长文档进行切片去重；优化 Prompt 长度。 |

## 10. 版本规划

### V1.0：基础问答版
- 核心：单轮对话 + 基础 RAG。
- 功能：文档上传、向量入库、简单问答、引用溯源。
- Agent 能力：仅包含“检索”工具。

### V2.0：多轮对话与工具增强
- 核心：多轮记忆 + 结构化数据查询。
- 功能：对话历史记忆、意图识别优化。
- Agent 能力：增加“数据库查询工具”（Text-to-SQL），支持回答“上个月销售额多少”。

### V3.0：自主 Agent 平台
- 核心：复杂任务拆解 + 自主规划。
- 功能：用户输入“帮我写周报”，Agent 自动检索本周日志、汇总数据、生成文档。
- Agent 能力：ReAct 架构全开，支持多步骤规划和自我修正。
