# Project Structure Constraints (必须严格遵守)

请严格遵循以下包结构进行架构设计，不要随意更改层级名称：

com.knowpivot.server
│
├─ interfaces # API层（对前端）
│ ├─ api # 存放 Controller，负责 HTTP 请求处理，参数校验
│ └─ vo # View Object，返回给前端的视图对象
│
├─ application # 用例编排层
│ ├─ service # Application Service，负责事务控制、领域服务编排
│ └─ dto # Data Transfer Object，各层间传输对象
│
├─ domain # 业务核心层
│ ├─ knowledge # 知识库领域模型
│ ├─ document # 文档领域模型
│ ├─ conversation # 会话/对话领域模型
│ # (注意：此层只包含业务逻辑、实体、领域服务接口、仓储接口，不包含任何技术实现)
│
├─ ai # ⭐ AI接入层（重点）
│ ├─ gateway # AI网关（统一入口，封装对 AI 能力的调用）
│ ├─ client # 客户端实现（如调用 Python 算法服务、OpenAI SDK 等）
│ ├─ model # AI请求/响应模型（Prompt模板、向量请求结构等）
│ └─ strategy # 多AI策略（策略模式，用于切换不同模型如 GPT/Claude/本地模型）
│
├─ infrastructure # 技术实现层
│ ├─ persistence # 持久化实现（Repository实现、Mapper/DAO、PO）
│ ├─ storage # 文件存储实现（MinIO/OSS本地存储）
│ └─ cache # 缓存实现
│
└─ common
└─ KnowPivotServerApplication.java

# Output Requirements (输出要求)

请按以下章节输出《系统架构设计文档》，内容需具体、落地，不要写空泛的理论：

## 1. 架构设计概览

- 简述系统整体架构风格（DDD + AI 网关）。
- 画出或描述请求从 "前端 -> Interfaces -> Application -> Domain/AI -> Infrastructure" 的完整链路流转图。

## 2. 分层架构详细设计

针对每一层（interfaces, application, domain, ai, infrastructure），结合需求文档，说明：

- **职责边界**：该层负责什么，不负责什么。
- **关键类设计**：列举几个核心类名（如 `KnowledgeController`, `KnowledgeApplicationService`, `KnowledgeRepository`），并说明其作用。

## 3. 核心 Domain 领域建模

针对需求中的核心业务（知识库、文档、会话），详细设计：

- **实体与值对象**：定义 `Knowledge`, `Document`, `Conversation` 的核心属性。
- **聚合根**：哪个实体是聚合根？如何保证一致性？
- **领域服务**：例如“文档解析服务”、“知识库检索服务”的逻辑归属。

## 4. AI 接入层架构设计（重点）

这是本系统的核心特色，请详细设计：

- **Gateway 设计**：如何通过 `ai.gateway` 屏蔽底层 AI 调用的复杂性？
- **Client 设计**：`ai.client` 如何与 Python 端或大模型 SDK 交互？
- **Strategy 策略模式**：`ai.strategy` 如何实现在不同大模型（如 GPT-4 vs 本地 Llama）之间动态切换？
- **Prompt 管理**：如何在 `ai.model` 中管理 Prompt 模板？

## 5. 数据流转设计（以“用户提问”为例）

请根据需求，描述一次典型的“用户基于知识库提问”的完整代码调用链：

- 从 `interfaces.api` 接收请求，到调用 `application.service` 编排，再到 `domain` 获取知识，通过 `ai.gateway` 获取答案，最后返回 `vo` 的全过程。

## 6. 技术选型与基础设施

- 列出推荐的技术栈（如 Spring Boot 版本、数据库选型、向量数据库选型、消息队列等），并简要说明理由。
