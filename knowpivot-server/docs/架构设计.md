# KnowPivot 系统架构设计文档

## 1. 架构设计概览

### 1.1 架构风格

本系统采用 **DDD（领域驱动设计）+ AI 网关** 的分层架构风格。

- **DDD 分层**：确保业务逻辑的高内聚，领域层聚焦于知识库、文档、会话的核心业务规则，不受技术实现细节污染。
- **AI 网关层**：作为连接 Java 业务系统与 Python AI 服务的“桥梁”，封装异构系统调用，实现模型切换、容错与降级，保证系统的可扩展性。

### 1.2 请求链路流转图

以下描述一次典型的“用户提问”请求流转：

```text
1. 前端
   ↓ HTTP Request (Query)
2. Interfaces Layer (interfaces.api)
   → ChatController 接收请求，参数校验，转换为 DTO
   ↓
3. Application Layer (application.service)
   → ChatApplicationService 开启事务
   → 权限校验 & 上下文组装
   ↓
4. Domain Layer (domain)
   → Conversation 聚合根：添加用户消息
   → MessageRepository：持久化用户消息
   → KnowledgeDomainService：校验知识库状态
   ↓
5. AI Layer (ai.gateway)
   → AgentGateway：封装 Prompt、历史记录、知识库 ID
   → Strategy：选择当前生效的 AI 模型策略
   → Client：发起 HTTP 调用到 Python 服务
   ↓ (Python 处理并返回)
6. Application Layer (回程)
   → 解析 AI 响应，更新 Token 配额
   → Conversation 聚合根：添加 AI 回复
   → MessageRepository：持久化 AI 回复
   ↓
7. Interfaces Layer
   → 封装 VO 对象
   ↓ HTTP Response
8. 前端
```

## 2. 分层架构详细设计

### 2.1 Interfaces 层 (API 层)

- **职责边界**：负责 HTTP 请求的接引入、参数基础校验（JSR-303）、身份认证信息提取、VO 与 DTO 的转换。**严禁**在此层编写业务逻辑。
- **关键类设计**：
  - `KnowledgeController`：处理知识库的创建、更新、删除、列表查询接口。
  - `DocumentController`：处理文档上传、解析状态查询、下载接口。
  - `ChatController`：处理用户发起对话、流式响应接口。
  - `KnowledgeVO` / `ChatVO`：返回给前端的视图对象，脱敏并格式化数据。

### 2.2 Application 层 (用例编排层)

- **职责边界**：负责事务控制、领域服务编排、权限校验（调用基础设施层）、DTO 转换。它是业务流程的指挥官，但不包含核心业务规则。
- **关键类设计**：
  - `KnowledgeApplicationService`：编排知识库的 CRUD 流程，如创建知识库时初始化默认配置。
  - `DocumentApplicationService`：编排文档上传流程 -> 保存文件 -> 发送消息到 MQ -> 返回受理结果。
  - `ChatApplicationService`：核心编排。负责组装 `Conversation` 上下文、调用 AI 网关、保存对话记录、扣减积分。

### 2.3 Domain 层 (业务核心层)

- **职责边界**：包含所有业务逻辑、规则校验、状态流转。不依赖任何技术框架，纯粹的业务模型。
- **关键类设计**：
  - `Knowledge` (Entity)：知识库实体，包含名称、描述、向量索引名等属性。
  - `Document` (Entity)：文档实体，包含 `DocumentStatus`（已上传、解析中、已索引、失败）、版本号等。
  - `Conversation` (Aggregate Root)：会话聚合根，管理 `Message` 列表，控制上下文窗口大小。
  - `KnowledgeRepository` (Interface)：定义知识库持久化接口，由 Infrastructure 实现。

### 2.4 AI 层 (AI 接入层)

- **职责边界**：隔离 Java 系统与外部 AI 服务。处理协议转换、模型适配、Prompt 模板管理。
- **关键类设计**：
  - `AgentGateway`：AI 能力的统一入口，对外暴露简单的业务语义方法（如 `chat(query, context)`）。
  - `HttpAgentClient`：基于 RestTemplate/WebClient 实现对 Python 服务的 HTTP 调用。
  - `ModelStrategy`：定义模型选择策略接口。

### 2.5 Infrastructure 层 (技术实现层)

- **职责边界**：实现技术细节，如数据库操作、缓存操作、文件存储、消息队列发送。
- **关键类设计**：
  - `KnowledgeRepositoryImpl`：实现 Domain 层接口，操作 MySQL。
  - `RedisVectorStore`：封装 Redis 向量检索命令，实现相似度搜索。
  - `MinioStorageClient`：封装 MinIO SDK，实现文件上传下载。

## 3. 核心 Domain 领域建模

### 3.1 实体与值对象设计

```java
    private List<MemberId> members;   // 授权成员列表
    private Timestamp createdAt;

    // 业务行为
    public void addMember(MemberId userId) { ... }
    public boolean hasPermission(MemberId userId) { ... }
}
```

#### B. `Document` (文档聚合根)

```java
public class Document {
    private DocumentId id;
    private KnowledgeId knowledgeId;
    private String fileName;
    private String storagePath;       // MinIO 路径
    private DocumentStatus status;    // 枚举：UPLOADED, PARSING, INDEXED, FAILED
    private int chunkCount;           // 切片数量

    // 业务行为：标记为解析中
    public void markAsParsing() {
        if (this.status != DocumentStatus.UPLOADED) {
            throw new IllegalStateException("文档状态异常");
        }
        this.status = DocumentStatus.PARSING;
    }
}
```

#### C. `Conversation` (会话聚合根)

```java
public class Conversation {
    private ConversationId id;
    private KnowledgeId currentKnowledgeId;
    private List<Message> messages;   // 实体列表
    private int tokenLimit;           // 上下文窗口限制

    // 业务行为：添加消息
    public void addMessage(Message message) {
        // 检查上下文长度，如果超限则进行摘要压缩或截断（领域逻辑）
        ensureContextWindow();
        this.messages.add(message);
    }

    private void ensureContextWindow() { ... }
}
```

### 3.2 领域服务

- **`DocumentParseService`**：虽然解析是在 Python 端，但 Java 端需要一个领域服务来决定解析策略（如：判断文件类型是 PDF 还是 Word，决定发送给 Python 的指令参数）。
- **`KnowledgeSearchService`**：定义检索逻辑接口（如 `searchTopK`），具体实现由 Infrastructure 层的 Redis 向量库完成，但业务规则（如相似度阈值过滤）应在 Domain 层定义。

## 4. AI 接入层架构设计（重点）

### 4.1 Gateway 设计

`ai.gateway` 屏蔽底层实现的复杂性，提供业务友好的接口。

```java
// com.knowpivot.server.ai.gateway.AgentGateway
public interface AgentGateway {
    // 执行对话，返回流式响应
    Flux<AgentResponse> chat(AgentContext context, String query);
}
```

- **实现类 `DefaultAgentGateway`**：
  1. 从 `ai.strategy` 获取当前生效的 Client。
  2. 调用 `ai.model.PromptFactory` 构建 Prompt。
  3. 调用 Client 发起请求。
  4. 捕获异常并降级（如调用备用模型）。

### 4.2 Client 设计

`ai.client` 负责底层通信，封装 Python Agent API 接口。

```java
// com.knowpivot.server.ai.client.PythonAgentClient
public class PythonAgentClient {

    // 对应需求文档中的 POST /agent/run
    public Flux<AgentResponse> runAgent(AgentRunRequest request) {
        // 使用 WebClient 发送 SSE 请求
        return webClient.post()
                .uri("/api/v1/agent/run")
                .header("X-Api-Key", apiKey)
                .bodyValue(request)
                .retrieve()
                .bodyToFlux(AgentResponse.class);
    }
}
```

### 4.3 Strategy 策略模式

`ai.strategy` 用于动态切换模型（如 GPT-4 vs 本地 Llama vs Qwen）。这对应需求文档中 Python 端的 `config.model` 字段，Java 端作为发起方需要控制策略。

```java
// com.knowpivot.server.ai.strategy.ModelStrategy

    String getModelCode();
    int getPriority();
}

// 实现类：QwenStrategy, GPTStrategy
// 业务层可根据用户等级或配置，选择不同的 Client 或设置不同的 Config 参数
```

## 5. 数据流转设计（以“用户提问”为例）

**场景**：用户在 Web 端输入“帮我总结项目A的风险点”，请求流经各层的详细过程。

1. **`interfaces.api.ChatController`**
   - 接收 `POST /api/v1/chat/message`。
   - 参数校验：检查内容长度、是否包含敏感词。
   - 调用 `ChatApplicationService.sendMessage(SendMsgCmd cmd)`。

2. **`application.service.ChatApplicationService`**
   - 开启数据库事务。
   - 调用 `conversationRepository.findById(cmd.getSessionId())` 获取会话聚合根。
   - 调用 `conversation.addMessage(userMessage)` 添加用户消息（领域行为）。
   - **构建 AI 请求上下文**：
     - 从 Redis 缓存获取最近 N 轮历史。
     - 从 DB 获取该知识库的配置。
   - **调用 AI 网关**：`agentGateway.chat(context, query)`。
   - **处理 AI 响应（流式）**：
     - 接收到 Token -> 拼接完整回复。
     - 接收到 `finish` 事件 -> 获取 Token 消耗。
   - **更新领域状态**：
     - 调用 `conversation.addMessage(aiMessage)` 添加 AI 回复。
     - 调用 `userAccountRepository.deductToken(userId, tokenCost)` 扣减配额。
   - 提交事务。

3. **`ai.gateway.DefaultAgentGateway`** (被 Application 调用)
   - 加载 Prompt 模板。
   - 封装 `AgentRunRequest` 对象（包含 session_id, query, history, config）。
   - 调用 `pythonAgentClient.runAgent(request)`。

4. **`infrastructure.persistence.MessageRepositoryImpl`**
   - 将 Domain 层的 Message 实体转换为 PO (Persistent Object)。
   - 执行 `INSERT INTO t_message ...`。

5. **返回前端**
   - Controller 返回 `Flux<ServerSentEvent<ChatVO>>`，实现流式打字机效果。

## 6. 技术选型与基础设施

### 6.1 核心技术栈

| 类别           | 技术选型                   | 选型理由                                                                                      |
| :------------- | :------------------------- | :-------------------------------------------------------------------------------------------- |
| **核心框架**   | Spring Boot 3.2.x + JDK 17 | 长期支持版本，Virtual Threads (虚拟线程) 支持更高效的并发处理，特别适合 AI 阻塞式调用场景。   |
| **ORM 框架**   | MyBatis-Plus               | 国内生态成熟，开发效率高，支持复杂查询拼接，符合 DDD Repository 接口映射习惯。                |
| **数据库**     | MySQL 8.0                  | 核心业务数据存储，支持 JSON 字段存储复杂配置。                                                |
| **向量数据库** | Redis Stack (RediSearch)   | **完全契合需求**。利用 Redis 原生向量检索能力，无需引入 Milvus 等重型中间件，降低运维成本。   |
| **文件存储**   | MinIO                      | 开源、高性能，兼容 S3 协议，适合私有化部署场景。                                              |
| **消息队列**   | Apache Kafka               | 用于文档解析的异步解耦。相比 RabbitMQ，Kafka 更适合处理大批量的日志类消息（如文档切片任务）。 |
| **前端通信**   | WebFlux (SSE)              | 实现流式响应的关键，避免阻塞 Servlet 线程池，提升系统吞吐量。                                 |

### 6.2 基础设施设计要点

- **Redis 向量索引**：需要在 `Infrastructure` 层封装 `FT.CREATE` 和 `FT.SEARCH` 命令，实现文档切片的向量化存储与检索。
- **异步处理**：文档上传后，通过 `KafkaTemplate` 发送消息，`DocumentConsumer` 监听并调用 Python 服务进行 Embedding，更新文档状态。
